# Amendment to Implementation Plan — Task 25.3

## 1. Статус

Данный документ вносит обязательные уточнения к Implementation Plan для Task 25.3.

Изменения являются обязательными для принятия реализации.

---

# 2. Policy по scenario_version

## 2.1 Принцип

После принятого решения об удалении служебных version-полей из parquet-артефактов A0.x:

* scenario_version
* etl_version
* component_algo_version
* qc_version

не должны присутствовать в сценарных таблицах.

## 2.2 Уточнение

* Повышение scenario_version допускается только во внутреннем pipeline logging.
* Версия не должна добавляться в parquet.
* GUI не должен отображать version-поля.

## 2.3 Запрет

Запрещено:

* поднимать версию исключительно из-за GUI-рефакторинга;
* использовать version-поля как часть структуры A0.x таблиц.

---

# 3. Запрет на схлопывание spatial dimension

## 3.1 Архитектурный инвариант

После перехода к C3 v4.0.1 spatial dimension является обязательной частью контракта.

A0.0 и A0.1:

* не имеют права агрегировать left/center/right в одну строку;
* не имеют права рассчитывать "общую" baseline или variability без spatial breakdown;
* не имеют права усреднять spatial dimension внутри C3.4.

## 3.2 Явное требование

Любая логика вида:

```
groupby(['subject_id','session_id'])
```

без учета `stimulus_location` — запрещена.

---

# 4. Контроль полноты триады location

## 4.1 Обязательная проверка

Для каждой session_id необходимо проверить:

```
set(stimulus_location) == {'left','center','right'}
```

## 4.2 Поведение при нарушении

Если триада неполная:

* выбрасывать ValueError;
* не заполнять отсутствующие поля нулями;
* не заменять на средние значения;
* не скрывать ошибку.

## 4.3 Fail-fast

Проверка должна выполняться до pivot.

---

# 5. Условие принятия реализации

Task 25.3 может быть принят только при:

1. Отсутствии version-полей в parquet.
2. Отсутствии схлопывания spatial dimension.
3. Наличии проверки полноты триады.
4. Отсутствии автоматического заполнения нулями.
