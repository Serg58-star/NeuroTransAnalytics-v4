# Task 29 — Stage 1: Масштабная инвариантность латентной геометрии

## Контекст

Данный Task реализует **ЭТАП 1 (Инвариантность к масштабированию и метрике)**
из документа *Exploratory Architecture Framework v4*.

Цель этапа — проверить, сохраняется ли латентная структура
при минимальных метрических преобразованиях данных.

Мы не меняем архитектуру признаков.
Мы меняем только масштаб представления.

---

# Предварительный шаг — стратегическое выравнивание

Перед началом вычислительных процедур GoAn обязан:

1. Ознакомиться с документом **Exploratory Architecture Framework v4.md**.
2. Проанализировать все этапы (1–9) как единую стратегию exploratory_lab.
3. Сопоставить запланированные преобразования с уже существующими инструментами проекта:

   * текущими скриптами PCA,
   * bootstrap-модулями,
   * split-half реализациями,
   * регрессионными процедурами,
   * продольным анализом,
   * существующей архитектурой exploratory_lab.
4. Сформировать краткий технический план реализации Stage 1:

   * какие модули будут переиспользованы,
   * какие функции требуют расширения,
   * какие элементы уже реализованы и могут быть параметризованы,
   * какие изменения минимальны и не нарушают архитектуру.

Принципиально:

* не создавать дублирующие скрипты,
* не нарушать архитектурную чистоту проекта,
* не выносить вычисления за пределы exploratory_lab,
* не вводить новые сущности без необходимости.

Отчёт о стратегическом плане должен предшествовать вычислительной части Task 29.

---

# Общая цель Task 29

Определить, является ли выявленная двумерная латентная геометрия:

* метрически зависимой,
* или порядково-инвариантной.

---

# Исходные данные

Используется текущий production-набор признаков,
на котором ранее была подтверждена двумерная структура.

Все расчёты выполняются:

* без изменения состава признаков,
* без удаления наблюдений,
* без агрегации полей зрения.

---

# Перечень преобразований

## 29.1 Log-transform

Для всех непрерывных RT-показателей:

* применить натуральный логарифм
* предварительно проверить отсутствие нулевых значений
* при необходимости использовать log(x + ε)

После преобразования выполнить:

* PCA
* Bootstrap PCA
* Split-half
* расчёт Between/Within

---

## 29.2 Box–Cox оптимизация

Для каждого непрерывного признака:

* определить оптимальный λ
* выполнить трансформацию
* зафиксировать найденные λ

После преобразования выполнить:

* PCA
* Bootstrap
* Split-half
* расчёт Between/Within

---

## 29.3 Rank-based PCA

* заменить значения признаков на ранги по всей выборке
* выполнить PCA
* выполнить Bootstrap
* выполнить Split-half
* рассчитать Between/Within

Цель: проверить порядковую инвариантность структуры.

---

## 29.4 Z-нормировка внутри субъекта

Для каждого subject_id:

Z = (X − mean_subject) / SD_subject

Цель:

* убрать абсолютную скорость
* сохранить относительный профиль

После преобразования выполнить полный набор метрик.

---

## 29.5 Z-нормировка внутри теста

Выполнить нормировку отдельно для:

* simple
* color_red
* shift

Сохранить структуру полей зрения.

После преобразования выполнить полный набор метрик.

# Методологические ограничения (обязательные к соблюдению)

Данный подпункт уточняет границы реализации Stage 1.

## 1. Набор признаков

Для всех преобразований используется **строго тот же production-набор из 7 признаков**,
на котором ранее была подтверждена устойчивая двумерная структура (Task 27).

Использование:

* 11 baseline-признаков,
* альтернативных агрегатов,
* расширенных или сокращённых наборов

— запрещено в рамках Stage 1.

Stage 1 тестирует метрическую инвариантность уже подтверждённой геометрии,
а не альтернативные представления.

---

## 2. Запрет на изменение генеративного контура признаков

Stage 1 является тестом масштабных преобразований.

Поэтому запрещено:

* вмешательство на trial-level данных,
* изменение логики TrialLevelDataLoader,
* изменение BaselineFeatureExtractor,
* пересборка агрегатов,
* изменение порядка вычислений признаков.

Все преобразования применяются **только к уже сформированному пространству признаков**.

---

## 3. Z-нормировка внутри теста (29.5)

Выполняется исключительно на уровне агрегированных признаков.

Группировка производится по принадлежности признаков к тестам:

* simple
* color_red
* shift

Нормировка trial-level данных в рамках Stage 1 запрещена.

---

## 4. Between/Within дисперсия

Метод расчёта отношения Between/Within обязан:

* полностью совпадать с реализацией Task 27.3J,
* использовать ту же формулу,
* применяться после каждой трансформации без изменения алгоритма.

Любое изменение метода расчёта запрещено.

---

## 5. Универсальность stability.py

Модуль geometry/stability.py должен быть:

* функционально изолированным,
* независимым от загрузки данных,
* пригодным для использования на последующих этапах Framework,
* не содержащим специфики Stage 1.

---

Нарушение любого из данных пунктов приводит к остановке Stage 1
и пересмотру реализации.

# Метрики, обязательные к отчёту

Для каждой трансформации сформировать таблицу:

| Transformation | Dim (λ>1) | PC1 % | Participation Ratio | Bootstrap PC1 SD | Split-half ΔPC1 | Between/Within |
| -------------- | --------- | ----- | ------------------- | ---------------- | --------------- | -------------- |

Где:

* Dim — число собственных значений > 1
* PC1 % — доля объяснённой дисперсии
* Participation Ratio — эффективная размерность
* Bootstrap PC1 SD — устойчивость первой компоненты
* Split-half ΔPC1 — инвариантность ориентации
* Between/Within — отношение межсубъектной к внутрисубъектной дисперсии

---

# Критерий закрытия Stage 1

Этап считается закрытым, если:

1. Размерность остаётся ≥ 2 при всех преобразованиях.
2. PC1 % не изменяется более чем на 5%.
3. Participation Ratio не снижается более чем на 0.1.
4. Between/Within остаётся существенно > 1.

Если хотя бы одно преобразование разрушает структуру,
это фиксируется как граница метрической валидности.

---

4. Указание, выполнены ли критерии закрытия.
